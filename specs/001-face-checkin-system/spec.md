# Feature Specification: 管理员场次人脸签到系统

**Feature Branch**: `[001-face-checkin-system]`  
**Created**: 2026-05-08  
**Status**: Draft  
**Input**: User description: "开发一整套包含管理员场次管理、二维码入场、免登录拍照签到、个人人脸照片库、个人签到记录、全局签到记录、异常处理和防重复控制的人脸签到系统。"

## Clarifications

### Session 2026-05-08

- Q: 普通用户、管理员、未登录扫码签到用户的权限边界如何划分？ → A: 普通用户登录后可访问首页、个人信息、自己的人脸照片库、扫码页和个人签到记录；管理员可访问全部管理页面与全局数据；未登录扫码签到用户只可访问扫码页、场次确认页、拍照签到页、签到结果页和结果查询接口，不能访问个人资料、个人照片库、个人历史记录、其他用户数据或任何管理员能力。
- Q: 普通用户上传人脸照片有哪些默认规则？ → A: 普通用户只能管理自己的照片；每人最多保留 5 张未删除照片；支持 JPEG、PNG、WEBP；单张图片上限 10 MB；无人脸、多张人脸、严重遮挡、严重模糊或质量不合格的图片直接拒绝进入可识别资料；照片处理采用“上传成功后异步检测和编码”的默认策略，并向用户展示待处理、处理中、可识别、失败等状态及失败原因。
- Q: 管理员管理人脸照片库有哪些默认规则？ → A: 管理员可查看所有用户的人脸照片和处理状态，可为任意用户上传、替换、删除照片，可查看检测或编码失败原因；管理员触发删除等关键动作必须记入审计日志；照片检测或编码失败后不提供管理员重检入口，由用户重新上传；删除照片不会回写修改历史签到记录，但会影响该用户未来的识别资料可用性。
- Q: 免登录扫码签到接口的边界是什么？ → A: 免登录签到必须从有效场次二维码 token 进入；免登录接口只允许获取当前场次必要信息、提交签到照片和查询本次处理结果；若二维码无效或场次不可签到，系统返回标准化状态码与原因；未登录签到成功时仅返回场次名称、签到状态、签到时间、脱敏展示名、attemptId 等必要字段，不返回个人资料、历史记录、照片库内容或管理数据。
- Q: 签到场次默认规则是什么？ → A: 管理员创建场次时必须填写名称、开始时间和截止时间，说明为可选；第一阶段不引入单独的迟到判定时间；场次状态固定为草稿、已发布、已关闭、已取消；只有已发布且当前时间落在开始时间与截止时间之间的场次允许签到；管理员可关闭、取消或重新生成二维码，重新生成后旧二维码入口失效。
- Q: 重复签到如何判定？ → A: 重复签到基于签到场次标识与识别出的用户身份联合判定，而不是基于登录用户；同一用户在同一场次只能拥有一条有效签到记录；重复扫码、重复拍照、重复提交统一返回 DUPLICATE_CHECKIN 或已签到提示；重复尝试会被记录为尝试或异常记录；系统需要在持久化层保证该唯一性约束。
- Q: 重复提交如何保持幂等？ → A: Flutter App 每次提交签到照片都生成 idempotencyKey；后端把该键的处理状态和最终结果保存在受控的幂等存储中；相同 idempotencyKey 的重复提交直接返回第一次处理结果，不重复做人脸识别，也不重复写入签到记录；幂等结果至少保留到场次截止后 24 小时。
- Q: 无人脸、多张人脸、低质量照片的默认处理规则是什么？ → A: 这类照片在用户照片上传阶段直接拒绝入库，在签到阶段不得生成有效签到记录；相关失败会形成签到尝试或异常记录；返回结果统一携带标准错误码；管理员可在异常记录中查看这些失败原因。
- Q: 签到成功、签到失败、签到处理中、已签到的返回模型如何划分？ → A: 对外主状态固定为 SUCCESS、FAILED、PROCESSING、DUPLICATE_CHECKIN；其中 PROCESSING 仅适用于异步排队、削峰或识别尚未完成的场景，并返回 attemptId 供 App 轮询查询结果；同时系统维护标准化结果码，至少覆盖 MANUAL_REVIEW、EXPIRED_SESSION、INVALID_IMAGE、NO_FACE、MULTIPLE_FACES、LOW_CONFIDENCE、FRS_TIMEOUT、FRS_RATE_LIMITED、FRS_ERROR 等细分结果；DUPLICATE_CHECKIN 独立于 FAILED，避免把已签到与真实失败混淆。
- Q: 个人记录与全局记录的可见范围如何定义？ → A: 普通用户登录后只能查看自己的签到记录，默认看不到识别置信度、原始签到照片或内部异常细节；管理员可查看全部用户记录，并按场次、用户、时间、状态筛选，也可查看异常签到尝试记录；管理员处理异常记录必须保留审计日志。
- Q: Flutter App 页面和设备验证边界如何分期？ → A: 第一阶段必须交付登录页、首页、个人信息页、人脸照片管理页、扫码页、场次确认页、拍照签到页、签到结果页、个人签到记录页、管理员场次管理页、管理员创建或编辑场次页、管理员二维码展示页、管理员场次签到记录页、管理员全局签到记录页、异常签到处理页；系统状态页和系统配置页列为第二阶段扩展；Windows 仅用于普通页面、列表、表单、状态流转和非硬件路由调试，摄像头权限、拍照、扫码、图片上传、网络访问、本地安全存储、未登录扫码签到链路、访问控制与结果展示必须在 Android 真机或模拟器验证。

### Session 2026-05-09

- Q: 第一版人脸识别能力来源与集成边界如何固化？ → A: 第一版必须通过 Spring Boot 后端统一封装华为云 FRS，并通过 FaceRecognitionProvider 抽象对业务层暴露 detect、enroll、search、compare 和 delete 等能力；Flutter App 不允许直接调用华为云 FRS；PostgreSQL 仍是唯一业务事实源；外部 faceId、faceSetName、similarity 和 externalRequestId 只作为外部引用、诊断或审计信息保存；本地开发与自动化测试默认使用 MockFaceRecognitionProvider；压力测试和常规开发验证走 Mock Provider 或 FRS stub，开发全部完成后再执行少量真实华为云人工验证。
- Q: 图片持久化和客户端访问边界如何固化？ → A: 人脸照片和签到照片如需持久化，必须由 Spring Boot 后端统一写入华为云 OBS，并在 PostgreSQL 中保存关联元数据；Flutter App 只能访问 Spring Boot 后端 API，不允许直接访问 PostgreSQL、Redis、RabbitMQ、华为云 OBS、华为云 FRS 或服务器文件系统；如果前端需要查看图片，只能使用后端生成的临时访问 URL 或后端代理访问。
- Q: JWT 失效和 Token 黑名单的默认规则是什么？ → A: 普通用户和管理员都支持注销；注销或被显式失效的 JWT 必须把其唯一标识写入 Redis Token 黑名单，并在剩余有效期内拒绝这些 Token 继续访问受保护接口。
- Q: 第一阶段签到识别与 RabbitMQ 的关系如何确定？ → A: 第一阶段签到主链路默认同步调用华为云 FRS 返回最终结果；RabbitMQ 在第一阶段强制用于人脸照片入库、失败重试和死信处理；签到削峰作为第二阶段扩展，但对外接口必须预留 PROCESSING 状态和结果查询契约，未来切换为异步时不破坏移动端交互模型。
- Q: 用户模型与唯一标识如何固定？ → A: 第一版不区分“人员”和“用户”，所有被识别、被签到、被管理员维护的主体统一为用户账户；系统使用唯一用户名区分用户和执行登录，不使用邮箱、手机号或其他外部标识作为主识别字段；管理员必须能够新增、编辑和停用用户。
- Q: 第一版签到识别是否需要活体检测？ → A: 不需要。第一版签到识别固定为“检测图片有效性 -> 搜索候选 -> 对候选执行人脸比对确认 -> 根据阈值判定结果”，不引入活体检测作为签到放行条件。
- Q: 第六张人脸照片和照片失败后的处理规则是什么？ → A: 每个用户最多保留 5 张未删除的人脸照片；Flutter App 和后端都必须在第 6 张上传时拒绝继续提交；如果照片检测或编码失败，系统返回失败原因，由用户自行删除旧照片并重新上传，不提供管理员重检入口。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 扫码完成人脸签到 (Priority: P1)

管理员发布一个可签到场次后，普通用户通过移动端扫描该场次二维码入口进入确认页，
即使当前未登录，也可以在该场次内拍照提交签到。系统识别出签到用户后，
要么生成该场次下唯一一条有效签到记录，要么返回明确失败原因、处理中状态或已签到提示。

**Why this priority**: 这是系统的核心价值交付。没有“发布场次 -> 扫码进入 -> 拍照识别 -> 生成记录”的闭环，
系统就不具备实际签到能力。

**Independent Test**: 创建并发布一个可签到场次，为某用户准备可识别的人脸照片后，
扫描场次二维码并提交签到照片。验证首次提交生成有效记录，重复提交不再生成第二条有效记录，
异步排队时能够返回处理中结果与查询引用，并且无效二维码、未开始、已截止、已关闭、已取消、
FRS 超时或限流等场景都能返回明确结果。

**Acceptance Scenarios**:

1. **Given** 某签到场次已发布且当前时间处于可签到窗口内，且用户的人脸资料可被识别，  
   **When** 用户扫描该场次二维码并提交合格的签到照片，  
   **Then** 系统返回签到成功结果，并在该场次下生成一条有效签到记录。
2. **Given** 用户已经在该签到场次中完成签到，  
   **When** 用户再次扫码或再次提交签到照片，  
   **Then** 系统返回已签到提示，且不会再生成新的有效签到记录。
3. **Given** 二维码无效、场次不存在、场次未开始、场次已截止、场次已关闭或场次已取消，  
   **When** 用户尝试进入或提交签到，  
   **Then** 系统拒绝签到，并返回与场次状态一致的失败原因。
4. **Given** 用户提交的签到照片无人脸、多张人脸、质量不合格、识别失败或识别置信度过低，  
   **When** 系统处理签到请求，  
   **Then** 系统返回明确失败原因，不生成有效签到记录，并在需要时记录异常或尝试信息。
5. **Given** 当前签到请求进入异步排队或识别尚未完成，  
   **When** 系统尚不能返回最终签到结论，  
   **Then** 系统返回处理中状态和查询引用，允许客户端后续查询最终结果。
6. **Given** 华为云 FRS 在签到处理过程中超时、限流或返回异常响应，  
   **When** 系统处理该签到请求，  
   **Then** 系统返回可追踪的失败或人工复核结果，不生成有效签到记录，且请求不会被静默丢弃。

### User Story 2 - 维护个人资料与人脸照片 (Priority: P2)

普通用户登录后，可以维护自己的用户名、密码、上传或替换自己的人脸照片，并查看自己的历史签到记录。
所有个人数据访问都必须严格限制在本人范围内；未登录扫码签到状态只能完成签到提交，
不能越权访问个人中心、人脸照片库或个人历史记录。照片提交成功后，
用户还能看到该照片的检测与识别准备状态，以及失败原因。

**Why this priority**: 可用的人脸资料是签到识别的前提，个人记录查询和自助资料维护是普通用户长期使用系统的基础体验。

**Independent Test**: 使用普通用户账户登录后，修改个人资料、上传或替换人脸照片、查看个人签到记录，
验证所有结果仅作用于本人，且单个用户不能超过允许的人脸照片数量上限。再在未登录扫码场景中尝试访问个人资料页、人脸照片页和个人签到记录页，
验证访问会被拒绝。

**Acceptance Scenarios**:

1. **Given** 普通用户已登录，  
   **When** 用户上传一张合格的人脸照片并提交，  
   **Then** 系统保存该照片资料，启动基于 FaceRecognitionProvider 的后续识别准备流程，
   并向用户展示该照片的处理状态。
2. **Given** 普通用户已登录，  
   **When** 用户修改个人资料、替换人脸照片或删除自己的照片，  
   **Then** 系统仅更新该用户自己的数据，且其他用户无法查看或修改这些内容。
3. **Given** 用户未登录且仅通过扫码进入签到场次，  
   **When** 该用户尝试访问个人资料、人脸照片库、个人签到记录或其他用户数据，  
   **Then** 系统拒绝访问，并要求先进入具备相应权限的登录状态。

### User Story 3 - 管理签到场次与异常记录 (Priority: P3)

管理员登录后，可以新增、编辑、停用用户，管理用户人脸照片库和签到场次，
并能查看场次签到情况、全局签到记录和异常签到记录。系统状态和系统配置属于第二阶段扩展，
不阻塞第一阶段签到闭环交付。管理员创建的每个签到场次都需要具备开始时间、截止时间、状态和
二维码入口，且关键管理操作需要保留审计痕迹。管理员还能查看照片处理失败原因、外部识别引用
状态或删除照片；照片失败后由用户重新上传，不提供管理员重检入口。

**Why this priority**: 管理员能力决定系统是否能被组织化使用，也是场次发布、异常纠正和结果追踪的控制中心。

**Independent Test**: 使用管理员账户新增、编辑并停用用户，再创建并发布签到场次，查看二维码、场次签到名单、全局签到记录和异常记录；
再用非管理员账户访问同类页面，验证只有管理员能执行这些操作，并且关键管理动作都会留下可追溯记录。

**Acceptance Scenarios**:

1. **Given** 管理员已登录，  
   **When** 管理员创建一个包含名称、说明、开始时间和截止时间的签到场次并发布，  
   **Then** 系统保存该场次并提供对应二维码供用户扫码进入。
2. **Given** 某签到场次已经产生签到结果，  
   **When** 管理员查看该场次签到用户列表或查看全局签到记录，  
   **Then** 管理员能够看到对应签到用户信息、签到状态和异常情况。
3. **Given** 存在识别失败、重复签到、高频请求或其他异常签到记录，  
   **When** 管理员进入异常处理流程，  
   **Then** 管理员可以查看异常详情、执行处理动作，并保留处理痕迹。
4. **Given** 管理员已登录，  
   **When** 管理员新增、编辑或停用某个用户，  
   **Then** 系统保存该用户状态变更，并且后续登录与管理权限遵循最新状态。
5. **Given** 某张照片的人脸准备或签到识别依赖的华为云 FRS 调用失败，  
   **When** 管理员查看照片详情或异常记录，  
   **Then** 管理员能够看到对应失败原因和外部请求诊断信息，并要求用户重新上传照片。
6. **Given** 非管理员用户尝试访问管理员页面或执行管理员操作，  
   **When** 系统校验其访问权限，  
   **Then** 系统拒绝访问或操作，不暴露管理员数据与管理能力。

### Edge Cases

- 上传的人脸照片不存在人脸时，系统如何返回标准错误码与失败原因？
- 上传的人脸照片包含多张人脸时，系统如何拒绝并提示重新上传？
- 上传或签到照片模糊、遮挡严重或质量不足时，系统如何返回可理解的失败原因？
- 人脸照片编码失败时，系统如何阻止该照片进入可识别资料范围？
- 用户达到可用照片数量上限时，系统如何阻止继续增加新的可用照片？
- 二维码无效、被篡改或已失效时，系统如何阻止进入场次？
- 场次处于草稿、已关闭或已取消状态时，系统如何区分各自的拒绝原因？
- 场次未开始、已截止、已关闭或已取消时，系统如何阻止签到？
- 同一用户在同一场次下重复扫码、重复拍照或重复提交时，系统如何保证只存在一条有效签到记录？
- 同一个签到请求被客户端重试时，系统如何保证结果一致且不产生重复有效记录？
- 系统进入异步排队或削峰处理时，客户端如何拿到处理中状态和后续查询引用？
- 未登录扫码状态尝试访问个人资料、个人照片库、个人签到记录或管理员页面时，系统如何拒绝？
- 普通用户尝试访问其他用户数据或执行管理员能力时，系统如何拒绝？
- 高频签到请求触发防刷控制时，系统如何反馈“请求过于频繁”而不是生成异常有效记录？
- 识别失败、低置信度、无法确定唯一身份时，系统如何返回失败结果并保留后续处理空间？
- 华为云 FRS 超时、限流或返回异常数据时，系统如何保证请求不被静默丢弃并保留人工复核空间？

## Constitution Alignment *(mandatory)*

- **Role Scope**: 本功能同时覆盖普通用户和管理员两类角色，不引入第三类角色。
- **Auth & Access**: 普通用户的个人资料、个人照片库和个人签到记录只允许本人在已登录状态下访问；
  管理员能力只允许管理员访问；扫码入场、场次确认、拍照签到、结果查看与结果查询可在未登录状态下进行，
  但仅限于当前二维码 token 对应的签到场次。
- **Data & Infra Boundary**: 所有个人资料、照片资料、场次信息、签到结果、异常记录和审计记录
  都通过后端受控能力访问；PostgreSQL 是唯一业务事实源；Redis 与 RabbitMQ 只承担
  支撑职责；华为云 OBS 只承担图片对象存储；外部 faceId、faceSetName、similarity 和
  externalRequestId 只作为引用或审计信息保存，不能替代本系统内部标识。
- **Face Pipeline Impact**: 本功能覆盖人脸照片采集、质量校验、基于华为云 FRS 的人脸检测、
  照片入库、人脸搜索、人脸比对、身份确认和签到结果生成全链路。
- **External Face Provider**: 本功能必须通过 FaceRecognitionProvider 统一封装华为云 FRS，
  Flutter App 不允许直接调用 FRS；Provider 超时、限流和异常响应必须映射为可追踪的结果。
- **Client Boundary**: Flutter App 只能访问 Spring Boot 后端 API，不允许直接访问
  PostgreSQL、Redis、RabbitMQ、华为云 OBS、华为云 FRS 或服务器文件系统。
- **Validation Target**: 需要覆盖角色访问边界、扫码签到链路、重复签到防护、幂等重试、
  处理中结果查询、异常场景处理、历史记录查询、外部识别服务失败处理，以及移动端扫码、拍照、
  上传、访问控制和结果展示的验证。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持普通用户登录。
- **FR-002**: 系统 MUST 支持管理员登录。
- **FR-003**: 系统 MUST 允许普通用户查看和维护自己的用户名与密码。
- **FR-004**: 系统 MUST 只允许普通用户上传、替换和删除自己的个人人脸照片。
- **FR-005**: 系统 MUST 将普通用户未删除的人脸照片数量上限限制为 5 张；Flutter App 和后端在第 6 张上传时都 MUST 拒绝继续提交；当没有至少一张可识别照片时，该用户不能被成功识别签到。
- **FR-006**: 系统 MUST 仅接受 JPEG、PNG 和 WEBP 格式的人脸照片，且单张图片大小不得超过 10 MB。
- **FR-007**: 系统 MUST 在人脸照片进入可识别资料范围前校验是否存在单张清晰且可用的人脸。
- **FR-008**: 系统 MUST 对通过校验的人脸照片启动异步识别准备流程，通过 FaceRecognitionProvider 执行外部检测与入库，并将成功结果关联到对应普通用户。
- **FR-009**: 系统 MUST 让普通用户看到自己每张人脸照片的处理状态和失败原因，至少覆盖待处理、处理中、可识别和处理失败四种状态。
- **FR-010**: 系统 MUST 允许管理员新增、编辑和停用用户账户。
- **FR-011**: 系统 MUST 以唯一用户名作为用户区分和登录标识，MUST NOT 使用邮箱、手机号或其他外部标识作为主识别字段。
- **FR-012**: 系统 MUST 允许管理员查看、上传、替换和删除任意用户的人脸照片，并查看这些照片的处理状态、失败原因和外部识别引用状态。
- **FR-013**: 系统 MUST 将管理员触发的用户状态变更、照片删除、场次状态变更、异常处理、配置调整和人工复核动作记入审计记录；照片失败后 MUST 要求用户重新上传，而不是提供管理员重检入口。
- **FR-014**: 系统 MUST 允许管理员创建签到场次，并要求场次至少保存名称、开始时间、截止时间、状态和创建人信息；场次说明为可选信息。
- **FR-015**: 系统 MUST 将第一阶段的签到场次状态限制为草稿、已发布、已关闭和已取消。
- **FR-016**: 系统 MUST 允许管理员发布、关闭或取消签到场次。
- **FR-017**: 系统 MUST 为每个签到场次提供一个二维码入口，且二维码内容不得直接暴露容易被猜测的顺序编号。
- **FR-018**: 系统 MUST 允许管理员重新生成某个签到场次的二维码入口；重新生成后，旧二维码入口必须失效。
- **FR-019**: 系统 MUST 允许移动端在扫码后获取对应场次的必要信息，用于确认是否进入正确场次。
- **FR-020**: 系统 MUST 只允许用户通过有效场次二维码 token 进入免登录签到链路。
- **FR-021**: 系统 MUST 允许用户在扫码进入指定场次后不登录直接拍照提交签到照片。
- **FR-022**: 系统 MUST 将未登录扫码状态的能力限制为场次确认、签到照片提交、签到结果查看和结果查询，禁止该状态访问个人资料、个人照片库、个人签到记录、其他用户数据或管理员功能。
- **FR-023**: 系统 MUST 在用户进入签到场次或提交签到时校验二维码有效性、场次存在性、场次发布状态、开始时间、截止时间以及场次是否已关闭或取消。
- **FR-024**: 系统 MUST 仅允许已发布且当前时间落在开始时间与截止时间之间的场次接受签到。
- **FR-025**: 系统 MUST 在处理签到照片时校验是否存在单张可识别的人脸，并识别无人脸、多张人脸和质量不合格等异常情况。
- **FR-026**: 系统 MUST 仅在签到照片通过校验后通过 FaceRecognitionProvider 先执行人脸搜索，再对候选结果执行人脸比对确认，并尝试匹配到唯一用户身份。
- **FR-027**: 系统 MUST 基于签到场次标识与识别出的用户身份联合判定重复签到，而不是基于当前登录用户判定。
- **FR-028**: 系统 MUST 在识别成功且该用户尚未完成当前场次签到时生成一条有效签到记录。
- **FR-029**: 系统 MUST 将有效签到记录关联到对应场次、识别出的用户身份、签到结果和签到时间。
- **FR-030**: 系统 MUST 在持久化结果层保证同一用户在同一签到场次下最多只存在一条有效签到记录。
- **FR-031**: 系统 MUST 在同一用户重复签到时返回 DUPLICATE_CHECKIN 或等价的已签到提示，并且不得生成新的有效签到记录。
- **FR-032**: 系统 MUST 要求移动端对每次签到照片提交携带唯一的 idempotencyKey。
- **FR-033**: 系统 MUST 对同一签到请求的重复提交保持幂等，使重试直接返回第一次处理结果，不再重复执行身份识别或再次调用外部识别 Provider，也不再写入新的有效签到记录。
- **FR-034**: 系统 MUST 将幂等结果至少保留到场次截止后 24 小时，以覆盖客户端重试和用户复查场景。
- **FR-035**: 系统 MUST 对过于频繁的签到请求实施限流，并返回明确失败结果。
- **FR-036**: 系统 MUST 在照片无人脸、多张人脸、质量不合格、识别失败、识别置信度过低、场次不可签到、二维码无效、FRS 超时、FRS 限流或 FRS 异常返回等情况下返回明确失败原因和统一错误码。
- **FR-037**: 系统 MUST 对外支持 SUCCESS、FAILED、PROCESSING 和 DUPLICATE_CHECKIN 四类主签到状态。
- **FR-038**: 系统 MUST 仅在请求进入排队、削峰或识别未完成时返回 PROCESSING 状态，并返回可用于结果查询的 attemptId 或等价引用。
- **FR-039**: 系统 MUST 允许移动端在收到 PROCESSING 状态后查询本次签到请求的最终结果。
- **FR-040**: 系统 MUST 允许普通用户在登录后查看自己的个人签到记录。
- **FR-041**: 系统 MUST 阻止普通用户访问其他用户的数据。
- **FR-042**: 系统 MUST 阻止非管理员访问管理员页面或执行管理员操作。
- **FR-043**: 系统 MUST 默认限制普通用户的个人签到记录视图仅展示自己的场次名称、签到时间、最终结果和必要说明，不展示原始签到照片、识别置信度或内部异常细节。
- **FR-044**: 系统 MUST 允许管理员查看某个签到场次下的签到用户列表。
- **FR-045**: 系统 MUST 允许管理员查看全局签到记录，并按场次、用户、时间范围和结果状态筛选。
- **FR-046**: 系统 MUST 允许管理员查看异常签到记录和签到尝试记录并执行处理动作。
- **FR-047**: 系统 MUST 在管理员处理异常签到时保留原始异常信息与处理结果之间的可追溯关系。
- **FR-048**: 系统 MUST 允许管理员查看系统状态，并且只能调整被产品明确标记为管理员可维护的系统配置项。
- **FR-049**: 系统 MUST 为移动端第一阶段提供登录页、首页、个人信息页、人脸照片管理页、扫码页、场次确认页、拍照签到页、签到结果页和个人签到记录页。
- **FR-050**: 系统 MUST 为管理员第一阶段提供场次管理页、场次创建或编辑页、二维码展示页、场次签到记录页、全局签到记录页和异常签到处理页。
- **FR-051**: 系统 MUST 将系统状态页和系统配置页定义为第二阶段扩展页面，不阻塞第一阶段的人脸签到闭环交付。
- **FR-052**: 系统 MUST 将返回给未登录签到用户的结果信息控制在必要范围内，仅可返回场次名称、签到状态、签到时间、脱敏展示名、attemptId 和用户理解本次结果所需说明，且不得返回个人资料、历史记录、照片库内容或管理员数据。
- **FR-053**: 系统 MUST 能够将重复签到尝试、识别失败尝试、被限流拦截的签到请求和其他被拒绝的免登录签到提交记录为异常或尝试记录，但这些记录不得升级为新的有效签到记录。
- **FR-054**: 系统 MUST 明确区分“签到失败”和“已签到”，不得把 DUPLICATE_CHECKIN 归并为通用失败。
- **FR-055**: 系统 MUST 要求摄像头权限、拍照能力、扫码能力、图片上传、网络访问、未登录扫码签到链路、本地安全存储、页面访问控制和所有签到结果展示状态在 Android 真机或模拟器上完成验证。
- **FR-056**: 系统 MUST 通过统一的 FaceRecognitionProvider 抽象封装 detect、enroll、search、compare 和 delete 等外部识别能力，业务模块不得直接调用厂商 SDK 或 REST API。
- **FR-057**: 系统 MUST 在第一版使用 HuaweiCloudFrsFaceRecognitionProvider 作为生产实现，并将 faceSetName、faceId、similarity 和 externalRequestId 仅作为外部引用、诊断或审计信息持久化。
- **FR-058**: 系统 MUST 为签到和照片处理维护标准化结果码，至少覆盖 MANUAL_REVIEW、EXPIRED_SESSION、INVALID_IMAGE、NO_FACE、MULTIPLE_FACES、LOW_CONFIDENCE、FRS_TIMEOUT、FRS_RATE_LIMITED 和 FRS_ERROR。
- **FR-059**: 系统 MUST 在华为云 FRS 超时、限流、异常响应或结果不确定时生成可查询的失败或人工复核结果，并记录可追踪的第三方调用日志，不得静默丢弃请求。
- **FR-060**: 系统 MUST 通过 RabbitMQ 对人脸照片入库执行异步处理，并支持失败重试与死信处理；签到识别第一阶段可以同步调用华为云 FRS，但 MUST 保留 PROCESSING 状态、attemptId 结果查询和未来切换为异步削峰时不破坏外部契约的扩展能力。
- **FR-061**: 系统 MUST 支持 MockFaceRecognitionProvider 或等价测试替身，以便本地开发、压力测试和自动化测试在未接入真实华为云 FRS 时仍可稳定运行。
- **FR-062**: 系统 MUST 在人脸照片和签到照片需要持久化时，通过 Spring Boot 后端统一写入华为云 OBS，并在 PostgreSQL 中保存 bucket、objectKey、contentType、sizeBytes、sha256 和 storageProvider 等对象元数据；OBS 不得作为业务事实源。
- **FR-063**: Flutter App MUST 只通过 Spring Boot 后端 API 访问业务能力，不得直接访问 PostgreSQL、Redis、RabbitMQ、华为云 OBS、华为云 FRS 或服务器文件系统。
- **FR-064**: 系统 MUST 支持普通用户和管理员注销，并将被注销或显式失效的 JWT 唯一标识写入 Redis Token 黑名单；黑名单中的 Token 在剩余有效期内 MUST 被受保护接口拒绝。

### Key Entities *(include if feature involves data)*

- **用户账户**: 表示系统内可以登录、被识别和被签到的唯一主体，包含唯一用户名、密码摘要、角色和状态。
- **人脸照片**: 表示普通用户或管理员维护的人脸图片资料，包含来源、状态、可用性信息、失败原因和所属用户。
- **图片对象元数据**: 表示存储在华为云 OBS 的人脸照片或签到照片对象引用，包含 bucket、objectKey、contentType、sizeBytes 和校验摘要，用于把对象存储与 PostgreSQL 业务事实关联起来。
- **可识别人脸资料**: 表示已经通过校验并可用于身份识别的人脸资料集合。
- **签到场次**: 表示一次由管理员创建和维护的签到活动，包含名称、说明、时间窗口、状态和创建人信息。
- **场次二维码入口**: 表示与签到场次绑定的扫码进入凭据，包含当前有效 token 与失效关系，用于把用户引导到指定场次。
- **签到请求**: 表示用户在某个场次中提交的一次签到尝试，包含照片、提交时间、idempotencyKey、attemptId 和处理结果。
- **有效签到记录**: 表示某用户在某场次中成功完成签到后的正式结果记录。
- **异常签到记录**: 表示识别失败、重复签到、被限流或其他异常情况下保留的处理记录。
- **外部人脸引用**: 表示本地用户或照片与华为云 FRS `faceSetName`、`faceId`、`similarity` 和外部请求标识之间的关联，仅用于识别引用、诊断和审计。
- **第三方识别调用审计**: 表示对华为云 FRS 的请求、响应、重试、超时、限流和失败诊断留痕。
- **审计记录**: 表示管理员关键操作的可追溯留痕，用于追踪管理行为和后续复核。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 95% 的管理员常规场次创建流程能够在 2 分钟内完成，并成功生成可用于扫码进入的场次入口。
- **SC-002**: 90% 的合格签到尝试能够在用户扫码后的 60 秒内返回明确结果。
- **SC-003**: 100% 的同用户同场次重复签到尝试都不会生成第二条有效签到记录。
- **SC-004**: 100% 的被拒绝签到尝试都能返回用户可理解的失败原因，并能区分场次不可用、照片问题、识别失败、权限不足或请求过频。
- **SC-005**: 100% 的管理员关键操作都能产生可审计记录，且非管理员对管理员能力的访问成功率为 0。
- **SC-006**: 95% 的普通用户照片上传后都能在 2 分钟内看到最终处理状态或明确失败原因。
- **SC-007**: 100% 的华为云 FRS 超时、限流或异常响应都能产生可追踪的失败或人工复核结果，不会静默丢失签到或照片处理请求。

## Assumptions

- 系统只包含普通用户和管理员两类角色，不包含教师、审核员或访客等其他角色。
- 普通用户账户与管理员账户都由系统内部管理；公开注册流程不在本次功能范围内；所有用户都以唯一用户名区分，不依赖邮箱或手机号登录。
- 第一版人脸识别能力固定通过 Spring Boot 后端统一封装华为云 FRS；图片对象持久化固定通过华为云 OBS；Flutter App 只访问 Spring Boot 后端 API，不直接访问 PostgreSQL、Redis、RabbitMQ、华为云 OBS、华为云 FRS 或服务器文件系统。
- 一个普通用户最多维护 5 张未删除的人脸照片；上传第 6 张时客户端和后端都会拒绝；只有通过校验并完成识别准备的照片会参与后续身份识别。
- 一个签到场次在任意时刻只对外提供一个有效扫码入口；如管理员重新生成入口，则旧入口按场次规则失效或停用。
- 一个有效签到记录只代表“某用户在某场次中成功签到一次”；后续重复尝试只会返回重复结果或形成异常记录。
- 管理员处理异常记录时不会抹除原始异常痕迹，而是在原始记录基础上追加处理结论。
- 第一阶段不单独区分迟到状态；只要当前时间落在场次开始时间与截止时间之间，签到即视为有效窗口内签到。
- 第一阶段默认采用同步签到识别主链路；PROCESSING 状态和结果查询接口作为后续异步削峰扩展的兼容契约保留。
- 第一阶段默认要求用户通过官方移动端完成扫码、拍照、上传和结果查看；离线签到不在本次范围内。
- 本地开发、压力测试与自动化测试默认依赖 MockFaceRecognitionProvider；真实华为云 FRS 调用放在开发全部完成后的少量人工验证中执行。
- Windows 桌面调试只用于普通页面、列表、表单、状态流转和非硬件路由验证；硬件与安全存储相关能力必须在 Android 设备环境验证。
